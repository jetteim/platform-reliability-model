# Incident To Postmortem To Learning

This scenario shows how an agent should transform incident evidence into postmortem learning and action items.

## Agent Inputs

- incident record
- timeline evidence
- deployment or change references
- affected entities
- observability evidence
- response communications
- resolution notes
- related incidents

## Agent Outputs

- normalized incident record
- impact and urgency assessment
- postmortem analysis
- timeline events
- contributing causes
- root cause categories
- lessons learned
- action items
- observability requests where needed

## Workflow

1. Normalize the incident record.
2. Confirm impact, urgency, and postmortem requirement.
3. Build a chronological timeline.
4. Preserve evidence before retention expires.
5. Document issue resolution.
6. Evaluate impact from user, service, team, and business perspectives.
7. Analyze trigger, enabling components, and response timeline.
8. Classify contributing causes.
9. Write lessons as reusable guidance for other teams.
10. Create action items that follow directly from analysis.

## Human Review Gates

- Confirm impact and urgency.
- Confirm timeline accuracy with responders.
- Confirm contributing causes are evidence-backed.
- Confirm lessons are not blame or vague advice.
- Confirm action items have owners and verification methods.

## Completion Criteria

- timeline is chronological
- evidence is preserved
- contributing causes explain trigger and response dynamics
- lessons are reusable
- action items improve reliability capabilities
