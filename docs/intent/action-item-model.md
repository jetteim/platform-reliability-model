# Action Item Model

Action items are the main output of incident aftercare.

## Required Properties

- source incident or postmortem
- evidence link
- contributing cause link
- improvement category
- owner
- priority
- target date or review date
- expected reliability effect
- verification method
- status

## Improvement Categories

- incident follow-up
- improve detection
- improve first response
- improve dispatch
- improve repair
- improve automatic mitigation
- improve graceful degradation
- improve dependency resilience
- improve release safety
- improve documentation or playbooks
- improve team or company process

## Rejection Criteria

Reject or rewrite action items that:

- say only "be more careful"
- assign blame
- do not follow from analysis
- have no owner
- have no verification method
- are too vague to schedule
